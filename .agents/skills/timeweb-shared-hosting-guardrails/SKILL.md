# Цель

Удерживать все технические решения в рамках shared hosting на Timeweb с PHP 8.5.

# Жесткие ограничения

- Не проектируй решение, требующее постоянных демонов.
- Не рассчитывай на supervisor.
- Не рассчитывай на production Node.js runtime.
- Не предлагай SSR на Node.
- Не делай архитектуру, которая требует Redis queue workers как обязательного условия для работы ядра.
- Не привязывай критический функционал к websocket-инфраструктуре.
- Не предлагай тяжелый media pipeline на сервере.

# Что предпочитать

- Laravel + MySQL.
- Blade + Livewire + Tailwind.
- Filament для admin/CRM.
- Scheduler через cron.
- Очереди только если есть безопасная деградация или можно запускать контролируемо.
- Сборка frontend assets локально или в CI с публикацией готовых assets.
- Хранение видео вне shared hosting.

# Правила проектирования

## Фоновые задачи
- Предпочитать cron + `php artisan schedule:run`.
- Любая фоновая логика должна иметь безопасный сценарий без long-running worker.
- Если предлагается queue, объясни:
  - зачем она нужна
  - что сломается без нее
  - есть ли fallback

## Медиа
- Не хранить тяжелые видеофайлы локально на хостинге без крайней необходимости.
- Использовать внешний `video_url`.
- Для изображений ограничить разумные размеры и форматы.
- Генерацию thumbnails делать только если это реально нужно и инфраструктурно допустимо.

## Деплой
- Учитывать схему Laravel на shared hosting:
  - document root на `public`
  - `storage:link`
  - корректные права
  - `.env`
  - миграции
  - кэш конфигов/маршрутов только если это безопасно для workflow

## Очереди и scheduler
- Если предлагаешь notifications, emails, sync tasks — укажи, как они будут работать без постоянного worker.
- Предпочитай:
  - cron-triggered commands
  - idempotent sync commands
  - safe retries

# Что проверять перед любым архитектурным предложением

Спроси себя:
1. Это точно совместимо с shared hosting?
2. Нужен ли здесь отдельный runtime?
3. Можно ли сделать проще на PHP?
4. Есть ли скрытая инфраструктурная зависимость?
5. Можно ли потом масштабировать без полного переписывания?

# Запрещенные решения по умолчанию

- Next.js SSR как основной фронт
- Nuxt SSR как основной фронт
- NestJS backend для MVP на этом хостинге
- отдельный websocket-сервер
- обязательный Redis как критическая зависимость
- видео-хранилище с обработкой на сервере
- сложные event-driven схемы без необходимости
